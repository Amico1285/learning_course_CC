# Будущие сессии

## Кейс 1. Дизайнер как владелец визуального слоя продукта

### Стилизация компонентов с помощью ИИ

#### Краткое описание кейса

В этом кейсе показывается, как дизайнер без навыков программирования берет на себя задачу приведения компонентов, написанных на чистом MUI, к внешнему виду дизайн-системы компании — без участия фронтенд-разработчика.

Для этого используется:
- Open Code (ИИ с доступом к репозиторию)
- Storybook для визуальной проверки компонентов
- MUI Theme и токены дизайн-системы
- работа напрямую с кодом компонентов (через ИИ)

В результате дизайнер:
- управляет внешним видом продукта на уровне кода
- сам вносит изменения в компоненты
- берет на себя ответственность за консистентность UI

#### Что именно хочется, чтобы было показано

**Общий процесс**
- с чего начинается задача
- как дизайнер объясняет ИИ, что нужно изменить
- как в целом устроен процесс работы

**Работа с дизайн-системой**
- как используются токены
- как происходит маппинг токенов на тему MUI
- где проходит граница между "дизайнерским" и "техническим"

**Контроль качества**
- как дизайнер понимает, что компонент сделан правильно
- как минимизируются риски сломать что-то в коде
- какие ограничения и проверки есть в процессе

**Встраивание в командный процесс**
- как такие изменения живут в гите
- как они ревьюятся
- как разработчики относятся к такому формату работы

#### Почему этот кейс важен для нас именно сейчас

Кейс особенно важен в контексте активного процесса создания и стилизации компонентов в соответствии с UI-китом и стайлгайдом в нашей компании.

Основная задача — найти способы ускорить и упростить процесс приведения компонентов к UI-киту и стайлгайду, не снижая качество и не создавая технический долг.

#### Что именно мы хотим найти в этом кейсе

Этот кейс рассматривается как источник практических ответов на вопросы:

**Как дизайнер может:**
- быстрее адаптировать компоненты под стиль продукта
- минимизировать количество итераций "дизайн - разработка - правки"
- брать на себя часть работы по визуальной настройке компонентов (или же всю работу по визуальной настройке)

**Как ИИ может быть встроен в процесс:**
- как помощник при работе с темами, токенами и стилями
- как переводчик между UI-китом и кодом компонентов

> ИИ — это главный исполнитель. Руками не делается вообще ничего.

## Кейс 2. Легковесные интерактивные прототипы как инструмент команды

### ИИ + Git + Replit

#### Краткое описание кейса

Этот кейс про сборку живых, кликабельных прототипов, которые:
- выглядят как реальный продукт
- используют реальные компоненты и паттерны
- можно быстро деплоить и показывать

В процессе используются:
- Claude Code с доступом к репозиторию
- Git как основа процесса
- Replit как среда сборки, деплоя и одновременно инструмент проектирования
- библиотека компонентов, экранов и паттернов

**Важно:** прототипами могут пользоваться не только дизайнеры, но и продакты, аналитики, другие участники команды.

Прототип — не замена Figma, а другой уровень прототипирования. Он позволяет:
- раньше находить проблемы во флоу
- лучше обсуждать сложную логику
- быстрее проверять гипотезы (демо клиентам / юзабилити-тестирования)

Показывает, как ИИ может быть частью общего продуктового инструментария, а не только дизайнерского.

#### Что именно хочется, чтобы было показано

**Архитектура прототипов**
- из чего состоит "основа" (компоненты, экраны, паттерны) и как ее подготовить
- как она переиспользуется
- что отличает это от "одноразового прототипа"

**Процесс сборки**
- как из готовых блоков собирается новый сценарий
- как в этом помогает ИИ
- какие задачи решаются быстрее, чем в Figma

**Работа со сложными флоу**
- где интерактивный прототип дает больше понимания
- какие механики сложно или невозможно адекватно проверить в статике

## Кейс 3. Техписатель и агентный AI

### Два занятия: от первых шагов до автоматизации

#### Текущая ситуация

Команда технических писателей (4 человека на 5 продуктов):
- Используют браузерное расширение (DeepSeek/ChatGPT) для понимания вопросов, но не для написания
- Драфты пишут вручную
- Документация хранится в Git (MD-файлы)
- Есть инструмент для перевода RU->EN, но качество английской документации низкое
- Гайдлайн по написанию документации в разработке (на основе гайдлайна Яндекса)
- Процесс ревью: техрайтеры проверяют друг друга
- Не используют AI для релиз-ноутов и whitepaper
- Нет системы сбора фидбэка о качестве документации

#### Проблемы, которые хотим решить

- Неконсистентная терминология между продуктами и статьями
- Низкое качество английской документации после автоперевода
- Waterfall-процесс: нет понимания, понятна ли документация пользователям
- Много ручной работы на релиз-ноуты и обновление документации

---

### Занятие 1: Агентный AI для техписателя — первые шаги

#### Краткое описание занятия

Введение в работу с агентным AI (Claude Code / Open Code) для технических писателей. Фокус на задачах, которые можно начать делать самостоятельно сразу после занятия.

Для этого используется:
- Open Code / Claude Code с доступом к репозиторию документации
- Глоссарий терминов (если есть) или его создание
- Гайдлайн по написанию документации

В результате техписатель:
- понимает разницу между браузерным чатом и агентом с доступом к репозиторию
- умеет проверять терминологическую консистентность
- умеет делать качественный перевод с пост-редактированием
- умеет проверять документ против гайдлайна

#### Что именно будет показано

**Ресёрч: как это делают другие**
- обзор гайдлайнов для техписателей (Google, Microsoft, Яндекс, Diataxis)
- как формализовать гайдлайн для автоматической проверки (Vale, LLM)
- сравнение подходов к переводу: DeepL vs LLM (Claude, GPT)
- как крупные open source проекты (Kubernetes, React) организуют локализацию

**Введение в агентный AI**
- что такое Claude Code / Open Code vs браузерные чаты
- почему агент + Git = мощнее, чем копипаст в ChatGPT
- демо: агент работает с документацией в репозитории

**Практика: проверка терминологии**
- как дать агенту глоссарий и документ
- как агент находит неконсистентное использование терминов
- как итеративно улучшать результат

**Практика: перевод с пост-редактированием**
- почему LLM лучше автоматических переводчиков для сложного контента
- workflow: перевод -> ревью -> правки
- как давать агенту контекст для лучшего перевода

**Практика: ревью против гайдлайна**
- как формализовать гайдлайн для агента
- проверка структуры, тона, форматирования
- что агент ловит хорошо, а что требует человека

#### Тизер: что дальше

- Автоматизация этих проверок в CI/CD
- Почему "просто добавить в пайплайн" не работает — нужен evaluation, эксперименты с моделями, prompt engineering

### Занятие 2: От драфтов до автоматизации

#### Краткое описание занятия

Продвинутые сценарии использования агентного AI: генерация контента, автоматизация рутины, построение пайплайнов. Показываем как возможности для самостоятельной работы, так и горизонт того, что требует экспертизы.

В результате техписатель:
- умеет генерировать драфты релиз-ноутов
- умеет генерировать драфты документации на новую фичу
- понимает, как агент может быть первым ревьюером
- видит перспективу автоматизации и понимает, что для этого нужно

#### Что именно будет показано

**Ресёрч: как это делают другие**
- Conventional Commits и автоматическая генерация changelog (semantic-release, release-please)
- AI-инструменты для релиз-ноутов (ReleasesNotes.dev, GenAIScript)
- агентные пайплайны для документации (RepoAgent, DocAider)
- как Stripe, Microsoft, Google собирают фидбэк о документации
- метрики качества документации (Task Completion Rate, CSAT)

**Генерация релиз-ноутов**
- откуда брать данные: changelog, PR descriptions, commit messages
- как агент трансформирует технические изменения в понятный текст
- практика: сгенерировать релиз-ноут для реального релиза

**Генерация драфтов документации**
- как дать агенту спецификацию фичи и получить структуру статьи
- итеративная доработка драфта
- где агент силен, а где нужен человек

**Агент как первый ревьюер**
- автоматическая проверка перед человеческим ревью
- что агент находит: опечатки, неконсистентность, нарушения гайдлайна
- как это экономит время команды

#### Горизонт: что требует серьезной работы

Показываем схему "идеального пайплайна":

```
PR merged -> Агент: извлечь изменения -> Агент: драфт релиз-ноутов
                                      -> Агент: обновить документацию
          -> Агент: проверить терминологию
          -> Агент: перевести на EN
          -> Агент: ревью против гайдлайна
          -> Человек: финальное ревью
```

**Почему это не делается за выходные:**

| Задача | Что нужно |
|--------|-----------|
| **Evaluation** | Как измерить, что агент хорошо перевел? Нужны метрики, тестовые наборы, baseline |
| **Выбор модели** | Claude vs GPT vs DeepL — разные результаты, разная стоимость. Тестирование на ваших данных |
| **Prompt engineering** | Первый промпт работает на 60%. Довести до 90% — десятки итераций |
| **Цепочки агентов** | Ошибка одного агента каскадом идет дальше. Нужна архитектура с проверками |
| **Интеграция** | Git hooks, CI/CD, права доступа, секреты, мониторинг |
| **Edge cases** | Документ на 50 страниц? Смешанный RU/EN? Нестандартный формат? |

> Простые вещи можно делать самостоятельно уже сегодня. Для построения production-ready пайплайна нужна экспертиза — исследование, эксперименты, настройка.
