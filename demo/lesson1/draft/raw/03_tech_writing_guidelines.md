# Гайдлайны для технических писателей: обзор ресурсов

## 1. Google Developer Documentation Style Guide

### Основные принципы

- **Голос и тон**: документация должна звучать как разговор с осведомленным другом, который понимает, что разработчик хочет сделать. Дружелюбно, уважительно и конверсационно.
- **Глобальная аудитория**: рекомендации учитывают людей, для которых английский не является родным языком, а также переводчиков.
- **Доступность**: руководство содержит принципы создания документации, доступной для людей с различными ограничениями.
- **Инклюзивный язык**: гендерно-нейтральный язык, избегание оскорбительных и дискриминирующих терминов.
- **Гибкость**: приоритет отдается ясности и консистентности для конкретной предметной области, даже если это означает отступление от общих рекомендаций.

### Ключевые техники письма

- Использование второго лица (you/вы)
- Настоящее время
- Активный залог
- Серийная запятая (Oxford comma)

## 2. Microsoft Writing Style Guide

### Основные принципы

- **Голос**: теплый и расслабленный, четкий и ясный, готовый помочь.
- **Краткость**: не тратить слова впустую. Избегать лишних квалификаторов. Короткие и лаконичные предложения.
- **Приоритизация**: самое важное — в начало.
- **Сканируемость**: использование секций для разбивки длинных процедур на управляемые группы шагов (не более 12 шагов в процедуре).
- **Ориентация на пользователя**: перед написанием четко определить, кто пользователь и какую задачу он пытается выполнить.
- **Эмпатия и доступность**: акцент на доступности, разнообразии и инклюзивности.

### Структура руководства

- Планирование контента и дизайна
- Грамматика и пунктуация
- Форматирование текста
- Выбор слов и акронимы
- Капитализация
- Адаптивный и сканируемый текст

## 4. Diataxis Documentation System

### Концепция

Diataxis (ранее известная как Divio Documentation System) — систематический подход к созданию документации, основанный на понимании потребностей пользователей.

### Четыре типа документации

Система организована вокруг двух осей:
- **Практика vs Теория**: пользователь хочет что-то сделать или что-то узнать?
- **Приобретение vs Применение**: пользователь приобретает новые навыки или применяет уже имеющиеся?

#### 1. Tutorials (Учебные пособия)
- Урок, который ведет студента за руку через опыт обучения
- Всегда практический: пользователь делает что-то под руководством инструктора
- Фокус на обучении, а не на выполнении реальной задачи

#### 2. How-To Guides (Практические руководства)
- Адресуют реальные цели или проблемы
- Предоставляют практические указания для конкретных ситуаций
- Рассчитаны на уже компетентного пользователя
- Содержат список шагов, которые нужно выполнить по порядку
- Похожи на рецепт: четкая цель и определенный конец

#### 3. Reference (Справочник)
- Единственная задача — описывать максимально ясно и полно
- Описание корректного использования, а не демонстрация достижения цели
- Техническая спецификация

#### 4. Explanation (Объяснение)
- Относится к области пропозиционального знания, а не действия
- Служит для изучения, а не для работы
- Дает общее понимание без конкретной цели
- Также называется "discussions" или "explainers"

## 5. Формализация гайдлайна для автоматической проверки AI-агентом

### Инструменты

#### Vale — линтер для прозы
- Open-source линтер для проверки документации
- Поддерживает форматы: Markdown, HTML, reStructuredText, AsciiDoc, DITA, XML
- Правила хранятся в YAML-файлах
- Можно настроить на существующие стайлгайды или создать свои правила

#### Конфигурация Vale
- Файл `.vale.ini` для настройки
- Указание пути к файлам стилей
- Установка уровня предупреждений (error, warning, suggestion)
- Настройка для разных форматов файлов (HTML, Markdown)

#### Готовые наборы правил
- Elastic style guide rules
- Datadog documentation style guide
- GitLab Vale rules
- Grafana Writers' Toolkit

### AI + Традиционные инструменты

- **LLM-styleguide-helper**: комбинирует Vale с AI для автоматического исправления нарушений стайлгайда
- **LintGPT от Optic**: автоматизация API Style Guides с использованием ChatGPT
- AI-ревью должно дополнять, а не заменять существующие инструменты
- Линтеры — для быстрой обратной связи по синтаксису и стилю
- LLM — для более глубокого контекстного анализа

### Принципы формализации правил

1. **Структурированный формат**: правила в YAML/JSON для машинной обработки
2. **Уровни серьезности**: error, warning, suggestion
3. **Паттерны и регулярные выражения**: для автоматического поиска нарушений
4. **Примеры правильного и неправильного использования**: для обучения AI
5. **Интеграция в CI/CD**: автоматическая проверка при деплое

## 6. Общие best practices для технических писателей

### Принципы письма

- **Ясность и краткость**: передавать сложную техническую информацию просто и прямо, без лишнего жаргона и многословия
- **Знание аудитории**: анализировать целевую аудиторию и соответственно выбирать стиль, тон, длину и формат
- **Документировать очевидное**: не предполагать, что читатель знает то же, что и автор

### Структура и форматирование

- Организовывать контент с четкими заголовками и структурированной разметкой
- Направления (инструкции) — в нумерованные списки: один шаг = одно действие
- Перечисления — в маркированные списки
- Сложные списки — в таблицы
- Использовать визуальные элементы: скриншоты и схемы иногда лучше длинных предложений

### Тон и стиль

- Прямой и нейтральный тон
- Без шуток, случайных эмодзи и GIF-анимаций
- Простой язык: читателю не нужен словарь для понимания документации

### Консистентность и терминология

- Создание и следование стандартизированному глоссарию
- Единообразное использование терминов, фраз и технического жаргона
- Минимизация недопонимания

### Тренды 2025

- AI и автоматизация для рутинных задач и повышения качества контента
- AI-инструменты для проверки, генерации контента и предиктивного анализа на основе поведения пользователей

## Выводы

1. **Единство подходов**: Google, Microsoft и Яндекс сходятся в принципах ясности, краткости, ориентации на пользователя и инклюзивности.

2. **Структурирование по Diataxis**: система четырех типов документации (tutorials, how-to, reference, explanation) помогает организовать контент в соответствии с потребностями пользователей на разных этапах.

3. **Автоматизация проверки**: Vale и подобные линтеры позволяют формализовать стайлгайд в машиночитаемые правила. Комбинация с AI дает возможность не только находить, но и исправлять нарушения.

4. **Формализация для AI**: для автоматической проверки AI-агентом гайдлайн должен содержать структурированные правила (YAML/JSON), примеры правильного и неправильного использования, уровни серьезности и возможность интеграции в CI/CD.

5. **Практический фокус**: лучшие практики подчеркивают важность понимания аудитории, использования простого языка, структурированной подачи информации и консистентной терминологии.
